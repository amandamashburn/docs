## What to expect

Everything in the Extended Mind System lives in a database. Before diving into any individual collection, it's worth taking the time to understand what databases actually are, why they're so useful for organizing information, and how to get the most out of them in Notion and Obsidian.

This is required reading before exploring any of the collections. The concepts here show up everywhere.

## What will be covered

**What a database is** – At its core, a database is just a structured collection of related items. Your reading list is a database. Your project list is a database. Once you start seeing information this way, it changes how you think about organizing everything.

**What metadata is** – Metadata is information *about* your information. The author of a book, the status of a project, the date you saved an article – these are all metadata. We'll cover what metadata fields are, how to choose them, and which standard fields show up across almost every use case in this system.

**Building views** – One of the most powerful features of both tools is the ability to look at the same database in completely different ways without changing the underlying data. We'll cover filtering, sorting, grouping, and calculations – including custom formulas – and how to build views that surface exactly what you need, when you need it.

**Pages within databases** – Each entry in a database is also a full page. That's where the real magic is. The database gives you structure and organization; the page gives you depth. A single entry can hold text, images, embedded videos, links, and more – all organized under the same metadata that makes it findable and sortable.

**Database templates** – Templates let you define a starting structure for every new entry. A new book always opens with the same metadata fields. A new project always starts with the same sections. This is one of the highest-leverage habits you can build – it eliminates friction and enforces consistency without any extra effort.

**Notion implementation** – How databases work natively in Notion, the different view types available, quirks to be aware of, and how linked databases differ from source databases.

**Obsidian implementation** – How [Bases](https://help.obsidian.md/bases) treats your Markdown notes as database records and their properties (stored as YAML frontmatter) as fields – giving you filterable, sortable, grouped views over your entire vault. We'll cover how Bases works, how it integrates with Obsidian's existing Properties system, and what the quirks are worth knowing about.

## Database defined

A database is an organized collection of structured information (often in rows and columns) that allows for effecient management, retrieval, and querying with the ability to build-in constraints that ensure data accuracy.

Whether you realize it or not, the majority of people interact with databases regularly. When you search for a product online, check your bank balance, or look up a flight, you're pulling information from a database. 

For more information see: https://grokipedia.com/page/Database

## Metadata defined

Metadta is information about information or data about data.
It is structured information that describes, explains, locates, or otherwise makes it easier to retrieve, use, or manage other data. It's information that describes characteristics about the primary data. 

Some common examples of metadata.

| Primary source | Metadata |
|----------------|----------|
| book | author, publisher, publication date, ISBN |
| photo | timestamp, location, photo device |
| email | sender, recipient(s), timestamp, subject |

For more information see: https://grokipedia.com/page/Metadata

## Organizing with databases v. organizing with folders

In both Notion and Obsidian, you have the option to use folders in lieu of databases. 
But I advocate using databases as a fundamental design choice.

This is a fundamental concept of organization.
Flat file system v. nested file system
Both consist of a collection of pages
But one method allows you to slice and dice with ease and layer in reporting elements. 
Introducing relations (for advanced use).

When you step back, you realize that our lives center around some commone high level themes and common collections. Those would normally live as folders.

So why organize information in databases? Before we answer that, let's look at how data has been traditionally organized – with file systems.

Think of the files on your computer. You create folders. Sometimes you nest folder. And then you add files within those folders. This system does work. But it is a relic of the physical world where a single physical sheet of paper can only be slotted into one file folder.

Computers obviously changed this but many have stuck with the file system because while databases have existed since the 1960s, they weren't the most accessible to use and had limitations. But tools such as Notion and Obsidian have changed this. They have made databases accessible for non-technical users. The power of databases comes from views.

With databases – you can filter, sort, and group your pages.

Let's use recipes as an example. You could create a recipe folder. And then sub folders around appetizers, salads, entrees, soups, and desserts.

But that is merely one attribute that can be assigned to a recipe. You could also categorize recipes by
- their predominent ingredient (e.g., beef, chicken, veg)
- cooking time
- skill level
- seasons
- equipment required
- where you found the recipe (e.g, family recipe, cooking magazine, online recipe)
- recipes you've cooked v. ones you want to try

If you have recipes categorized by meal type but then want to view say only your family recipes that have been passed down or want to see dishes with season ingredients for Spring, you have no good way of viewing them within your file system (unless you happen to be very diligent about tagging).

Now, if you were to create a database of recipes – you can now have a page that contains the recipe, but then you can have additional properties (e.g., metadata) for each of the items listed above.

Then you could build specific views based on that metadata. Perhaps you create a view grouped by meal type and then another view grouped by season. Or perhaps, you want to add a tag called 'Thanksgiving' so you can build a view off all your go to reciipes for your annual Thanksiving day meal. 

As you can see, a database allows you to build different views based on what you want and need to see while all the underlying pages remain in a single collection. 

And views are extremely straight forward to create. So it's easy to spin them up for a permanent view you want to work in or you could easily create a temporary view for the task at hand and delete it when done.

Databases make it far easier to query and manipulate your pages / files. You don't have to limit yourself to a single heirarchy.

This is why the extended mind is organized around databases that I call collections. Through years of experimentation I've identified some high level buckets that your life's information can belong to. Those make up the core collection (e.g., people, places, things to name a few). Then there are additional reference collection items that get a bit more detailed (quotes, books, family history notes, etc).

The idea is to keep like documents with like.

And, another benefit of databases, should you ever get to this place is the ability to create relationships between databases. This is a more advanced topic that will not be covered here, but the point is, a database sets you up to scale and develop more complex relationships between pages should the need arise in time.
