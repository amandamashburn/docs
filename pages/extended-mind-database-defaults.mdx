Throughout my years of using Notion, I have developed certain preferences for how I stand up and configure my databases.
I like my databases to be consistent from one to the next. So the following lays out the standard setup, fields, views, and templates I use.
Rather than create a database from scratch each time, I have a standard database template where I maintain all these configurations. Whenever I need to create a new database, I duplicate the template and layer in additional modifications for the specific collection.
So you'll find throughout the collection descriptions on this site, I do not address the standard setup each time. I point out the additional layers to the standard setup.
As always, working from a template saves time (you aren't clicking 100 configurations to stand up a database) and it minimizes decision fatigue to keep databases consistent across the board.

Notion gives a ton of options in their databases. Which is great. So experiment. Learn what you like. Then create a standard template for yourself that you can duplicate and build upon when you want to spin up a new database.

Notion also gives the option to describe the database you want. I rarely use it because I keep my DBs on the simple side, but it's there to help if you'd like.

One downfall I consistenly see across people building their own systems is that they spend an immense amount of time setting up and tinkering with their system and trying to make it perfect from day one. They end up spending cycles on planning and configurations instead of living life and using the system to capture details of life. So, with that, if you are new and starting out. Use the default template I have provided or create your own. But keep it super simple when it comes to database views and properties. And add additional properties and views as you find you actually need them. Stated another way, don't try to build out a perfect and elaborate system from day one. Let your system naturally evolve with use.

## Standard metadata
A common behavior I observe in Notion spaces is for people to go hog wild with metadata fields. And while some collections warrant a variety of fields, most don't. You can organize your database and create a variety of views with the following fields. I suggest living with the least amount of database fields as possible and only add one if it is absolutely necessary.

### Name
Required field.

### Created on

Property type: Date

Date format: Full date

Time format: 24 hour

Having the creation date on display is incredibly helpful to know when you encountered a topic that you wanted to learn about, had an idea for a project, noted a topic you wanted to formulate an opinion on, etc. This is one of those fields that becomes more valuable as the years go by because you can begin to see clusters or long stretches where you were or were not logging certain types of information and topics. Which is a good mirror for the ebbs and flows of life (e.g., your knowledge bank reflects when you were circling a particualar theme), or, conversely a reflection of consistency for something like a daily log.

### Last edited

Property type: Date

Date format: Full date

Time format: 24 hour

Last edited is a fun one. So say you create a new page on year. And then 7 days or 7 months later go to udpate it. It's becomes a reflection of the longevity of an item when compared to the created on date. But also, it shows which pages you are interacting with the most recently.
I like to set a database to to sort by the last edited date in descneding order. That way, the items I've interacted sits at the top of the page. It naturally creates a list attuned to where my current focus is.

### Tags

Property type: Multi-select

Date format: Full date

Time format: 24 hour

One thing I go with my tags: I set all tags to a single color – yellow. When I first began using tags I would assign different colors to each tag. But as my tags grew beyond available colors, I realized I was thinking way too much and way too often about what tag should be what color. So I said enough with that and defaulted to a single color to simplify things. Occasionally I'll use a seconday color (grey) but only if it's really needed. Not only does it simplify things, but I find it keeps the entire look and feel of the space quite consistent which I find overall more visually appealing than a bunch of colors. (You'll note I do the similar with page icons – more on that below).

Tags is one place where I really like to use AI. I like to turn on 'AI autofill' as well as 'Generate new options'. Additionally I'll often at the following note on 'What to generate':

*Read the page title and content, then identify high level themes it covers. Prefer broad category tags over specific details, and choose fewer tags if unsure.*

The reason I like to use AI generated tags here is one it cuts down on my thinking and AI will sometimes suggest an applicable category I may not have considered on my own.

When it comes to tags, like databases, I try to keep the scope broad. I don't want to constantly be fiddling with tags or have the list get cluttered with hyper-specific items I'll only use a handful of times. A general rule of thumb is to think 'would I want to still use this tag 5 years from now?'.

Another thing I do with tags is set the property to sort the tags automatically. This helps with quick selection.

Mulit-select is used to assign multiple categories to a single item.


## Standard views

The following views are standard in all my databases. Should I require additional views, I duplicate the table and modify the layout, properties displayed, sorts, filters, and groupings. The main table view always remains intact with more granular views if necessary.

### Table
- For each database I keep a primary view in table format that display all metadata fields and pages within a database. This is so I can see the full contents and elements of the database. 
- Sort by Last edited (Descending)
- Hide vertical lines
- Show page icon
- Wrap all fields
- Open pages in side peek

### Gallery
Additionally I like to view pages as a gallery. It feels like staring at a nice grid of index cards. Same as the table view I like to Display all items sorted by Lasted Edit (descending).

- For this view, I opt not to display all properties. Instead I usually list the Last edited and Tag(s) fields.
- show page icon
- wrap content
- open page in side peek
- card preview: page content
- card size: large

### TODO
The TODO view is the table view with a filter applied to only display all pages that contain TODO in the title.

Tip: In notion, if you create a new page from this view, it will automatically update the page title to start with TODO.

## Standard page template
- serif font 
- dark grey page icon: again – reduce a decision and improve visual consistency across your space (some people love to use a different emoji but I find it makes a Notion space feel too busy).
- now there are times I may want to call attention to a particular page for whatever reason. for this task i'll update a page icon to a yellow star to visually highlight it.
- turn off comments at the top of the page if it's just using the space
- create the template. set it as the defualt. helps speed up the page creation process.

## Preferred database settings

- dark grey book icon (to indicate a collection of pages)
- black header to visually indicate I'm in a database
- turn off divider lines in tables
- side peek pages for all database views
- disable show vertical lines
- Add a count to the name field to show how many entries are in the database.
